Key reuse across protocols

One key. SSH, GPG, and Git.

Your OpenPGP authentication subkey can act as an SSH key. One key pair covers server login, commit signing, and GitHub — on both Windows and Linux.

Architecture

How it works

OpenPGP keys are structured as a primary (master) key plus subkeys, each with a specific capability flag: [S]ign, [E]ncrypt, [A]uthenticate, or [C]ertify. The [A] authenticate subkey is the bridge to SSH. When gpg-agent has SSH support enabled, it presents auth subkeys to any tool that speaks the SSH agent protocol — including ssh, git, VS Code remote, and GitHub CLI.

Primary key [C]

Certify only

Signs other keys and User IDs. Keep this offline when possible. Its fingerprint is your permanent identity.

Subkey [S]

Signing

Signs documents, commits, tags, and email. This is what GitHub shows as "Verified" on commits.

Subkey [E]

Encryption

Decrypts messages and files encrypted to your public key. Never used for SSH.

Subkey [A]

Authentication ← SSH

The subkey that becomes your SSH key. gpg-agent exposes it over the SSH agent socket.

gpg-agent

The bridge

Acts as both a GPG agent and an SSH agent when enable-ssh-support is set. One daemon, two sockets.

sshcontrol

Which keys are exposed

Only keygrips listed in ~/.gnupg/sshcontrol are offered to SSH clients. You control the list.

Key point

You do not convert a GPG key into an SSH key. You add an auth subkey and teach gpg-agent to speak SSH agent protocol. The private key material never leaves gpg-agent.

Fingerprint vs. keygrip — what's the difference?

Fingerprint

A hash of the full OpenPGP public key packet. This is your external identity — what you share with people, publish to keyservers, and verify in person at a key signing party. Every subkey has its own fingerprint, but you normally refer to the primary key's fingerprint.

# Primary key fingerprint (40 hex chars):
7DDC4865B5008DDE36C6B9A4FB113D111B055C9B

# How gpg displays it:
gpg -K
sec  ed25519 2026-04-27 [C]
     7DDC4865...1B055C9B

Keygrip

An internal handle derived from the raw key parameters (not the full OpenPGP packet). Used by gpg-agent to locate key material in its keybox and to decide which keys to expose over SSH via sshcontrol. The keygrip of the same mathematical key is identical across different GnuPG versions and machines.

# Auth subkey keygrip (40 hex chars):
57EEA3A4A6025A3BBFBAEC28ADD2ACA995BC185B

# How gpg displays it:
gpg --with-keygrip -K
ssb  ed25519 [A]
     Keygrip = 57EEA3A4...95BC185B
Rule of thumb

Use the fingerprint in commands that act on a key (gpg --export, git config user.signingkey, keyserver lookups). Use the keygrip only when configuring what the agent exposes — specifically in sshcontrol. You will rarely need a keygrip outside of that one file.

Setup

Prerequisites

Linux / macOS

GnuPG 2.2 or later

Most distributions ship a recent version. Verify with gpg --version. On macOS, install via Homebrew: brew install gnupg.

gpg --version
# Should show GnuPG 2.2.x or 2.4.x
Windows

Gpg4win (includes GnuPG + Kleopatra)

Download from gpg4win.org. Includes GnuPG, Kleopatra (GUI), and the agent. Prefer version 4.3.0 or later for better OpenSSH pipe support.

gpg --version
# Run in PowerShell or CMD after install
Both platforms

An existing OpenPGP key

You need at least a primary key with a User ID. If you do not have one yet, create one with gpg --full-generate-key and choose Ed25519. See the workflow guide for a complete setup path.

Windows + WSL

Optional: WSL2 bridge tools

To share the Windows gpg-agent with WSL2, you will need npiperelay.exe on your Windows PATH and socat inside WSL (sudo apt install socat). Or run an independent GnuPG in WSL (simpler).

Step zero

Add an authentication subkey

If your key already has an [A] subkey, skip to the platform sections. Run gpg -K --with-keygrip and look for [A] in the subkey list.

1Open the key editor

Replace FINGERPRINT with your full 40-character primary key fingerprint. The --expert flag unlocks the capability-selection menu.

gpg --expert --edit-key FINGERPRINT

2Add the subkey

At the gpg> prompt, enter addkey. Follow the menu to select an algorithm and capabilities:

gpg> addkey
Please select what kind of key you want:
   (3) DSA (sign only)
   (4) RSA (sign only)
   (5) Elgamal (encrypt only)
   (6) RSA (encrypt or sign)
   (7) DSA (set your own capabilities)
   (8) RSA (set your own capabilities)
  (10) ECC (sign only)
  (11) ECC (set your own capabilities)   ← choose this
  (12) ECC (encrypt only)
  (13) Existing key
Your selection? 11

Toggle capabilities so only Authenticate is on. Sign and Encrypt should be off:

Possible actions for this ECC key: Sign Authenticate
Current allowed actions: Sign

   (S) Toggle the sign capability
   (A) Toggle the authenticate capability
   (Q) Finished

Your selection? S   # turn Sign OFF
Your selection? A   # turn Authenticate ON
Your selection? Q   # done

Choose the curve. Ed25519 is recommended:

Please select which elliptic curve you want:
   (1) Curve 25519 *default*
   (2) Curve 448
   (3) NIST P-256
   ...
Your selection? 1

Set an expiry date (1–2 years is reasonable). It can be extended later without changing the key material.

3Save and verify

gpg> save

Confirm the new subkey appears:

gpg --with-keygrip -K FINGERPRINT

You should see an ssb line tagged [A] with its own Keygrip = line. Copy that keygrip — you will need it shortly.

Backup first

Adding a subkey changes your key. Export and back up your secret key before and after: gpg --armor --export-secret-keys FINGERPRINT > mykey-backup.asc. Store the file encrypted, offline.

Platform

LinuxEnable gpg-agent SSH mode

These steps work on Debian, Ubuntu, Fedora, Arch, and macOS. They configure the already-running gpg-agent to also handle SSH authentication requests.

1Enable SSH support in gpg-agent

mkdir -p ~/.gnupg
echo "enable-ssh-support" >> ~/.gnupg/gpg-agent.conf
chmod 700 ~/.gnupg

Restart the agent to apply the change:

gpgconf --kill gpg-agent
# gpg-agent restarts automatically on next gpg use

2Set SSH_AUTH_SOCK in your shell

Add this to your ~/.bashrc, ~/.zshrc, or ~/.profile. It replaces the default ssh-agent socket with gpg-agent's SSH socket:

export SSH_AUTH_SOCK=$(gpgconf --list-dirs agent-ssh-socket)
Disable the default ssh-agent

If your desktop environment or ~/.bashrc already starts ssh-agent, disable it. Having two agents fighting over SSH_AUTH_SOCK causes unpredictable failures. On GNOME, run gnome-session-properties and disable the SSH agent launcher if present.

Reload your shell config: source ~/.bashrc

3Register the auth subkey

Get the keygrip of your [A] subkey:

gpg --with-keygrip -K FINGERPRINT
# Look for the ssb line tagged [A] and its Keygrip value
# Example Keygrip: A3B4C5D6E7F8...

Add it to sshcontrol (one keygrip per line):

echo "A3B4C5D6E7F8..." >> ~/.gnupg/sshcontrol

The optional second column is a cache TTL in seconds. Leave it empty or set to 0 to always require passphrase confirmation.

4Verify the key is available to SSH

ssh-add -L
# Should print your public key in SSH format, e.g.:
# sk-ssh-ed25519@openssh.com AAAA... (none)

If you see your key, gpg-agent is acting as the SSH agent. If the list is empty, check that the keygrip is in ~/.gnupg/sshcontrol and the agent has been restarted.

5Export the SSH public key

This exports the auth subkey in OpenSSH public key format, ready to paste into authorized_keys or GitHub:

gpg --export-ssh-key FINGERPRINT

Save to a file for convenience:

gpg --export-ssh-key FINGERPRINT > ~/.ssh/id_gpg_auth.pub
cat ~/.ssh/id_gpg_auth.pub

6Add to remote authorized_keys

Copy your SSH public key to a server:

ssh-copy-id -i ~/.ssh/id_gpg_auth.pub user@server
# or manually:
cat ~/.ssh/id_gpg_auth.pub | ssh user@server "cat >> ~/.ssh/authorized_keys"

Test the connection:

ssh user@server

The agent will prompt for your GPG passphrase on first use (and cache it according to default-cache-ttl in gpg-agent.conf).

systemd users

On systemd-based systems, you can manage gpg-agent as a user service. The socket path is stable: gpgconf --list-dirs agent-ssh-socket always returns the right path regardless of session type.

Platform

WindowsEnable gpg-agent SSH mode

Windows uses named pipes instead of Unix sockets, and GnuPG 2.4+ supports the Windows OpenSSH named pipe directly. The steps below cover both native Windows OpenSSH and WSL2.

1Create gpg-agent.conf

GnuPG does not create this file automatically. The directory exists after installing Gpg4win, but the file must be created manually. Run this in PowerShell to create it with the required settings in one step:

@"
enable-ssh-support
enable-win32-openssh-support
"@ | Set-Content "$env:APPDATA\gnupg\gpg-agent.conf" -Encoding UTF8

To verify the file was created correctly:

Get-Content "$env:APPDATA\gnupg\gpg-agent.conf"

enable-win32-openssh-support creates a named pipe at \\.\pipe\openssh-ssh-agent that the Windows OpenSSH client reads automatically. Requires GnuPG 2.4.0 or later (included in Gpg4win 4.2+).

2Register the auth subkey

Get the keygrip of your [A] subkey. Pass your primary key fingerprint — gpg will list all subkeys:

gpg --with-keygrip -K 7DDC4865...1B055C9B

sec#  ed25519 2026-04-27 [C] [expires: 2028-04-26]
      7DDC4865B5008DDE36C6B9A4FB113D111B055C9B
      Keygrip = 1BE21959...5FCF399B
uid           [ultimate] James Aaron McKeeth <j**@m******.org>
ssb   ed25519 2026-04-27 [S] [expires: 2027-04-27]
      Keygrip = 0EAFF66B...3451B2EC
ssb   cv25519 2026-04-27 [E] [expires: 2027-04-27]
      Keygrip = BE8C2DE2...46905506
ssb   ed25519 2026-04-27 [A] [expires: 2027-04-27]
      Keygrip = 57EEA3A4...95BC185B   ← copy this one

The [A] subkey's keygrip is what goes in sshcontrol. Add it (the command creates the file if it does not exist):

Add-Content "$env:APPDATA\gnupg\sshcontrol" "57EEA3A4A6025A3BBFBAEC28ADD2ACA995BC185B"

Verify: Get-Content "$env:APPDATA\gnupg\sshcontrol" should show exactly one keygrip.

3Restart the agent and verify

Disable the Windows OpenSSH Authentication Agent service first

If the built-in OpenSSH Authentication Agent Windows service is running, it owns the \\.\pipe\openssh-ssh-agent named pipe and blocks gpg-agent from using it. Open Services (Win+R → services.msc), find OpenSSH Authentication Agent, set Startup type to Disabled, and click Stop. GnuPG's agent will take over the pipe after it restarts.

Kill the currently running agent so it picks up the new config on next start:

gpgconf --kill gpg-agent
gpg -K   # triggers agent restart

Check that SSH can see the key:

ssh-add -L
# Expected output:
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... (none)

If the list is empty, confirm sshcontrol contains the keygrip and that the agent restarted: gpg-agent --daemon --sh should print its PID.

4Export the SSH public key

gpg --export-ssh-key FINGERPRINT

Copy the output to clipboard or save to a file for use with GitHub and servers.

5WSL2 — share the Windows gpg-agent

WSL2 runs a separate Linux kernel and cannot access Windows named pipes directly. Two options:

Option A — Independent GnuPG in WSL (simpler)

Install GnuPG inside WSL, import your key material there, and follow the Linux steps inside WSL. You maintain two copies of the key, but everything works natively.

# Inside WSL:
sudo apt install gnupg
# Transfer key (see Sync section)
gpg --import mykey.asc

Option B — Bridge Windows agent into WSL

Uses npiperelay.exe (on Windows PATH) and socat (inside WSL) to relay the Windows named pipe to a Unix socket in WSL.

# Inside WSL — install socat:
sudo apt install socat

# Add to ~/.bashrc:
SSH_RELAY_SOCK="$HOME/.gnupg/S.gpg-agent.ssh"
if ! pgrep -f "npiperelay.exe" > /dev/null 2>&1; then
  rm -f "$SSH_RELAY_SOCK"
  (setsid nohup socat \
    UNIX-LISTEN:"$SSH_RELAY_SOCK",fork \
    EXEC:"npiperelay.exe -ei -s //./pipe/openssh-ssh-agent",nofork \
    > /dev/null 2>&1 &)
fi
export SSH_AUTH_SOCK="$SSH_RELAY_SOCK"

Reload: source ~/.bashrc. Then verify with ssh-add -L.

Windows clients

WindowsPuTTY and WinSCP

PuTTY and WinSCP do not use the Windows OpenSSH named pipe. Instead they communicate with Pageant — PuTTY's SSH agent — via a Windows shared-memory protocol. GnuPG's enable-putty-support makes gpg-agent speak that protocol, so PuTTY and WinSCP pick up your GPG-backed key automatically without Pageant ever being launched.

1Add Pageant support to gpg-agent.conf

Open %APPDATA%\gnupg\gpg-agent.conf and add the line. If you already have the file from the OpenSSH steps, append to it — both options can coexist:

@"
enable-ssh-support
enable-win32-openssh-support
enable-putty-support
"@ | Set-Content "$env:APPDATA\gnupg\gpg-agent.conf" -Encoding UTF8

Verify:

Get-Content "$env:APPDATA\gnupg\gpg-agent.conf"
# enable-ssh-support
# enable-win32-openssh-support
# enable-putty-support

2Register your auth subkey (if not already done)

Same as the OpenSSH steps — the keygrip goes into sshcontrol. Skip this if you already completed the Windows OpenSSH section:

Add-Content "$env:APPDATA\gnupg\sshcontrol" "57EEA3A4A6025A3BBFBAEC28ADD2ACA995BC185B"

3Restart gpg-agent

gpgconf --kill gpg-agent
gpg -K   # restarts the agent with the updated config

Do not start Pageant separately. gpg-agent now answers Pageant protocol requests itself. Running both causes conflicts.

4Test with PuTTY

First, get the SSH public key and add it to the server's authorized_keys (or GitHub — see the GitHub section):

gpg --export-ssh-key 7DDC4865...1B055C9B

Open a PuTTY session. Under Connection → SSH → Auth → Credentials, leave the private key field blank — PuTTY will request keys from the agent automatically. Connect to your server:

# Or test via plink (PuTTY's command-line SSH):
plink -v user@yourserver.example.com

PuTTY should authenticate without prompting for a private key file. The GPG passphrase prompt from gpg-agent (via pinentry) will appear instead.

PuTTY key format note

PuTTY natively uses .ppk files, but when using an agent (Pageant or gpg-agent with Pageant support) no .ppk file is needed or used. The agent provides the key directly. If PuTTY asks for a key file, it means agent negotiation failed — re-check that gpg-agent restarted after editing gpg-agent.conf.

5WinSCP: use the agent for file transfers

WinSCP supports SFTP and SCP over SSH. It uses the same Pageant shared-memory protocol as PuTTY, so once gpg-agent has enable-putty-support enabled, WinSCP picks it up automatically.

Verify Pageant agent auth is enabled in WinSCP:

  1. Open WinSCP → New Session or edit an existing saved session.
  2. In the session editor, go to Advanced… → SSH → Authentication.
  3. Ensure "Attempt to use Pageant agent for authentication" is checked (it is on by default).
  4. Leave the private key file field empty — the agent provides the key.
  5. Click OK, then connect. WinSCP will query gpg-agent via the Pageant protocol.
# WinSCP also ships a command-line tool (winscp.com):
winscp.com /command "open sftp://user@yourserver.example.com/" "ls" "exit"
WinSCP and the OpenSSH agent

WinSCP 5.21+ can also use the Windows OpenSSH named pipe (\\.\pipe\openssh-ssh-agent) if configured under Advanced → SSH → Authentication → Agent forwarding. If you have both enable-win32-openssh-support and enable-putty-support active, either path works — use whichever is already configured in your WinSCP sessions. More detail in the WinSCP authentication docs.

Service

GitHub: SSH auth + commit signing

GitHub uses your OpenPGP key in two separate roles: SSH authentication (cloning and pushing via git@github.com) and commit verification (the "Verified" badge on commits). Both use the same key, added through different settings screens.

1Add your SSH public key to GitHub

Export the auth subkey in SSH format:

gpg --export-ssh-key FINGERPRINT

Go to github.com/settings/keys → New SSH key. Set Key type to Authentication Key. Paste the output and save.

Test the connection:

ssh -T git@github.com
# Hi username! You've successfully authenticated...

Configure git to use SSH for GitHub:

git config --global url."git@github.com:".insteadOf "https://github.com/"

2Add your GPG public key to GitHub

Export the full public key (the primary key, not just the auth subkey):

gpg --armor --export FINGERPRINT

Go to github.com/settings/keys → New GPG key. Paste the armored public key block and save.

Match email addresses

GitHub links your GPG key to commits by matching the email in your commits against a verified email address on your GitHub account that also appears in the key's User IDs. Your commit author email (user.email in git config) must appear in one of your key's UIDs and be verified on GitHub.

3Verify the setup

Make a signed commit on any repo and push. The commit should show a green Verified badge on GitHub within a few seconds.

git commit --gpg-sign -m "test: verify GPG signing"
git log --show-signature -1

Configuration

Git commit and tag signing

Configure git to sign every commit and tag automatically. These settings go in your global git config (~/.gitconfig) and apply to all repositories on that machine.

  1. 1

    Set your signing key

    Use the full fingerprint of your primary key. Git will find the right signing subkey automatically.

    git config --global user.signingkey FINGERPRINT
  2. 2

    Enable automatic commit signing

    git config --global commit.gpgsign true
git config --global tag.gpgsign true
  3. 3

    Point git at the right GPG binary (Windows)

    On Windows, git may look for gpg in a different location. Check with where gpg in PowerShell and set the path explicitly if needed:

    git config --global gpg.program "C:\Program Files (x86)\GnuPG\bin\gpg.exe"

    In WSL, git uses the WSL gpg binary — no extra config needed if GnuPG is installed there.

  4. 4

    Verify a signed commit

    git log --show-signature -1
# Should show: gpg: Good signature from "Your Name <you@example.com>"
  5. 5

    Optional: SSH signing instead of GPG signing

    Git 2.34+ can sign commits with an SSH key directly (without GPG). This uses the auth subkey's SSH public key as the signing credential. It is simpler to configure but produces a different signature format — GPG-signed commits are more widely understood by forges. To use SSH signing:

    git config --global gpg.format ssh
git config --global user.signingkey ~/.ssh/id_gpg_auth.pub

    And add the key as a Signing Key (not Authentication Key) on GitHub under SSH and GPG keys.

Key management

Sync across machines

The private key material lives in gpg-agent's keybox and must be explicitly exported to move between machines. The public key travels freely. Never copy the raw keybox directory — always use GPG's own import/export.

1Export key material from the source machine

# Export secret key (includes all subkeys):
gpg --armor --export-secret-keys FINGERPRINT > mykey-secret.asc

# Export public key (safe to distribute):
gpg --armor --export FINGERPRINT > mykey-public.asc
Protect the secret key file

The .asc file containing your secret key should be transferred over an encrypted channel (SCP, SFTP, encrypted USB). Delete it from both ends once the import is done. The passphrase protects the key material inside the file, but the file itself should not linger on disk.

2Transfer to the destination machine

# Secure copy over SSH:
scp mykey-secret.asc user@destination:~/mykey-secret.asc

Or use an encrypted USB drive. After copying, import on the destination:

gpg --import mykey-secret.asc
gpg --import mykey-public.asc   # usually already included above

3Set ultimate trust on the destination

After importing your own key, set the trust level to ultimate so GPG treats it as your own key:

gpg --edit-key FINGERPRINT
gpg> trust
Your decision? 5   # I trust ultimately
gpg> quit

4Re-register the auth subkey

On the new machine, add the auth subkey's keygrip to sshcontrol again (the keygrip is the same as on the source machine):

gpg --with-keygrip -K FINGERPRINT
# Copy the [A] keygrip

# Linux:
echo "KEYGRIP" >> ~/.gnupg/sshcontrol

# Windows:
echo KEYGRIP >> %APPDATA%\gnupg\sshcontrol

5Clean up

# Delete the exported secret key file from both machines:
# Linux / WSL:
shred -u mykey-secret.asc   # or: rm mykey-secret.asc

# PowerShell (Windows):
Remove-Item mykey-secret.asc

Fixes

Troubleshooting

ssh-add -L returns "The agent has no identities"

Cause: The keygrip is missing from sshcontrol, or SSH_AUTH_SOCK points to the wrong agent.

Fix: Verify echo $SSH_AUTH_SOCK matches gpgconf --list-dirs agent-ssh-socket. Verify the keygrip is in ~/.gnupg/sshcontrol. Restart the agent: gpgconf --kill gpg-agent.

"Permission denied (publickey)" when connecting to a server or GitHub

Cause: The server's authorized_keys does not contain the key gpg-agent is offering, or the wrong agent is running.

Fix: Run ssh -v git@github.com and look for which key is being offered. Compare with gpg --export-ssh-key FINGERPRINT output and what is in the server's authorized_keys. They must match exactly.

Commits are not showing as "Verified" on GitHub

Cause: The commit author email does not match a verified email on GitHub that also appears in a key UID, or the GPG public key is not uploaded to GitHub.

Fix: Check git config user.email. Verify that email is both in your key's UIDs (gpg -K) and verified under GitHub Settings → Emails. Re-upload the public key after adding new UIDs.

Windows: ssh-add -L fails with "Error connecting to agent"

Cause: The Windows OpenSSH Authentication Agent service is running and owns the named pipe, blocking gpg-agent.

Fix: Open services.msc, find OpenSSH Authentication Agent, stop it, and set startup type to Disabled. Then restart gpg-agent: gpgconf --kill gpg-agent.

WSL: SSH key not found after setting up the socat bridge

Cause: npiperelay.exe is not on the Windows PATH visible from WSL, socat is not installed, or a stale socket file is blocking the relay.

Fix: In WSL, run npiperelay.exe --help to confirm it is accessible. Run rm -f "$SSH_RELAY_SOCK" to clear any stale socket, then source ~/.bashrc to restart the relay.

gpg-agent asks for passphrase on every git push

Cause: The cache TTL is set to zero or the agent restarts between operations.

Fix: Add cache TTL settings to gpg-agent.conf. Values are in seconds:

default-cache-ttl 3600
max-cache-ttl 86400

Restart the agent after editing: gpgconf --kill gpg-agent.

gpg --export-ssh-key returns nothing

Cause: The key has no [A] subkey. Only auth subkeys are exported by this command.

Fix: Follow the auth subkey creation steps to add one. Verify afterward: gpg -K should show ssb ... [A].

Git signing fails with "error: gpg failed to sign the data"

Cause: gpg.program in git config points to the wrong binary, or the pinentry program cannot open a dialog (common in headless / SSH sessions).

Fix: For headless sessions, set pinentry to console mode in gpg-agent.conf: pinentry-program /usr/bin/pinentry-curses. On Windows, ensure gpg.program in ~/.gitconfig points to the Gpg4win gpg.exe. Test manually: echo "test" | gpg --clearsign.
